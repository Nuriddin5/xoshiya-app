# Manual Task Report: Fix Slow First Recorder Start

## Problem
Pressing `Start` for the first recording could take 30 seconds or more before the recorder became active.

## Root Cause
- `AudioRecorder.start()` awaited `window.studyCapture.validateRubaiRuntime()` before opening the desktop audio stream.
- `validateRubaiRuntime()` runs the heavy Rubai Python dependency probe by spawning Python and importing `faster_whisper` and `ctranslate2`.
- That dependency probe is expensive on cold start and does not need to block the UI from starting local capture.

## Fix
- Replaced the recorder start-time `validateRubaiRuntime()` call with `getRubaiRuntimeStatus()`.
- `getRubaiRuntimeStatus()` performs the lightweight file readiness check and avoids the Python import probe.
- Kept the deeper Rubai validation path available in the main process for actual worker/transcription startup.
- Added timing logs around recorder start phases:
  - `[recorder] start requested`
  - `[recorder] ASR file readiness checked`
  - `[recorder] desktop audio stream acquired`
  - `[recorder] media recorder started`

## Expected Behavior
- First `Start` click should move to recording quickly after the lightweight readiness and desktop stream checks.
- If another delay remains, the new logs identify whether it is ASR file readiness, desktop stream acquisition, or MediaRecorder startup.
- First transcription can still take extra time when the Rubai worker cold-starts, but that no longer blocks the recording start button path.
