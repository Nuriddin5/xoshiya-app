# Manual Task Report: Fix Slow App Startup

## Problem Description
The app was experiencing a slow startup, staying on the "Starting..." (Loading settings...) screen for more than 15 seconds. This prevented users from quickly accessing settings, books, or history while the Rubai ASR runtime was being validated.

## Root Cause Analysis
- The preload bridge `getStartupState` method was awaiting both `getSettings` and `getRubaiRuntimeStatus` in a `Promise.all`.
- `getRubaiRuntimeStatus` performs multiple asynchronous disk operations (`stat` and `access`) to verify the Python runtime, worker script, and model files.
- The renderer's `App` component was conditionally rendering nothing but a loading message until the entire `startupState` (including Rubai status) was resolved.
- There was no timeout mechanism for the bridge calls, potentially allowing a hang if disk I/O was extremely slow.

## Changes Implemented

### 1. Data Model Update
- Modified `StudyCaptureStartupState` in `src/shared/types.ts` to make `rubaiRuntime` nullable. This allows the startup state to be considered "ready" for the UI even if the ASR status is still being fetched.

### 2. Preload Bridge Optimization
- Modified `loadStartupState` in `src/preload/preload.ts` to return immediately after fetching settings, leaving `rubaiRuntime` as `null`.

### 3. Renderer Initialization Improvements
- Updated `App.tsx` to:
  - Display the UI as soon as settings and basic readiness are loaded.
  - Fetch Rubai runtime status asynchronously after the initial render if it's missing from the startup state.
  - Add a 5-second safety timeout message for the `getStartupState` bridge call to prevent silent hanging on the loading screen.
  - Show a neutral "ASR loading" state while Rubai validation is in progress.
  - Convert background Rubai status fetch failures into a visible unavailable state instead of leaving the UI stuck in loading.

### 4. Dashboard Enhancements
- Updated `Dashboard.tsx` to handle `rubaiRuntime` being `null`.
- The dashboard remains accessible while Rubai status is still loading.
- ASR-dependent actions are marked as blocked until Rubai readiness is confirmed.
- Improved the side panel to show either `Loading...`, `Ready`, or an actionable unavailable message.

### 5. Recorder Guardrail Fix
- Review found that the first patch did not actually disable recording start while Rubai status was loading or unavailable.
- Updated `RecorderControls.tsx` so the start action is blocked until both setup readiness and Rubai readiness are complete, while stop behavior remains available for active sessions.

### 6. Backend Caching
- Added a `cachedRubaiPaths` variable in `src/main/rubai-runner.ts` to avoid repeated disk scans for the project root and runtime files during status checks.

## Review Findings
- `src/renderer/App.tsx`: background `getRubaiRuntimeStatus()` failures were only logged to the console, leaving the UI in a permanent neutral loading state with no actionable feedback.
- `src/renderer/components/RecorderControls.tsx`: the prior report claimed recording controls were blocked during ASR loading, but the start button was still enabled whenever a desktop source was selected.
